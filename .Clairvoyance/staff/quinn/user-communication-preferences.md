# User Communication Preferences

> Last updated: 2026-04-22
> Author: Quinn

## Summary

Morten is not a developer and prefers explanations at the architecture/product level. He thinks well in terms of systems, responsibilities, workflows, and tradeoffs, but technical implementation detail can be overwhelming.

Preferred tone: plain language, architecture-first, friendly coworker style, light dry humor allowed.

## Guidance

- Explain technical work in plain language first.
- Use architecture-level framing: what part of the system does what, how information moves, what decision matters.
- Avoid code jargon unless it is necessary, and define it briefly when used.
- For implementation tasks, summarize what changed and why it matters rather than walking through low-level mechanics.
- Ask before going deep into technical details.
- Keep a friendly coworker tone with occasional dry humor when it fits Morten's mood.
